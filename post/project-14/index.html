Problem Introduction

This project contains mimicked user behavior on the Starbucks rewards mobile application. In this app, Starbucks sends out offers to its customers, like BOGO (buy one, get one free) or a discount. Different users get different offers. So the main goal of this project was to figure out which offers succeed the most to which types of customers.

Data Description:

There are three main data sets that we will be working with for this project. They are Portfolio, Profile, and Transcript. Each dataset contains the following information:

Portfolio:

The porfolio dataset contains information on the offers, such as:

  • id (string) - offer id
  • offer_type (string) - type of offer ie BOGO, discount, informational
  • difficulty (int) - minimum required spend to complete an offer
  • reward (int) - reward given for completing an offer
  • duration (int) - time for offer to be open, in days
  • channels (list of strings)

Profile:

The profile dataset contains information about the customer, such as:

  • age (int) - age of the customer
  • became_member_on (int) - date when customer created an app account
  • gender (str) - gender of the customer (note some entries contain ‘O’ for other rather than M or F)
  • id (str) - customer id
  • income (float) - customer’s income

Transcript:

The transcript dataset contains information about what kind of activity was performed on an offer, such as:

  • event (str) - record description (ie transaction, offer received, offer viewed, etc.)
  • person (str) - customer id
  • time (int) - time in hours since start of test. The data begins at time t=0
  • value - (dict of strings) - either an offer id or transaction amount depending on the record

Strategy to Solve the Problem

After viewing what each dataset had to offer, the first thing I wanted to do was to start cleaning the datasets and changing them into a format that would seem appropriate for what I wanted to do. For the Portfolio, I thought it would be important to create categorical columns for the offer type and channel columns.

As for the Profile, it was necessary to reformat the became_member_on column, and create age groups and income classes in order to create visualizations easier to understand later on in the project.

The Transcript dataset required something interesting in order to understand the data a bit better. I wanted to separate the offers and transactions into two different dataframes. This allowed it to be easier to organize the information presented in the offer types and the transactions.

Metrics

Here, I would like to discuss how we will be measuring the performance of the classification models that will be made We will be using the following:

  • F1 Score: This is the weighted average of the Precision and Recall of the model.
  • Precision shows the ratio of the amount of True Positives to the sum of the amount of True Positives and the amount of False Positives. This ultimately will tell us: out of all of those that were predicted as positive, how many of them are actually positive.
  • Recall shows us the ratio of the amount of True Positives to the sum of the amount of True Positives and the amount of False Negatives. The will tell us how many of the actual positive values our model has found through labeling it as a True Positive.
  • Accuracy will tell us how accurate our model is by dividing the number of correct prediction by the total number of predictions.

Finding Successful Offers

In order to find whether or not an offer was successful, we need to go through the data that is given to us in a particular manner.

For every unique customer id, we are going to get the offers and transactions associated with that id and all of the information that goes along with those that are found in the Profile and Portfolio dataframes.

Once we have that information for that one unique customer id, we will go through each row of information to see whether or not the offer was completed within the time window where the offer is valid. If this is true, then we can say the offer was successful. We do this for all offers, for all unique customer ids. Once this column is obtained, we can start visualizing the data.

For income classes, we see that offers are more successful for upper and above average customers, and more unsuccessful for average and lower income customers.

Modelling

For the model, we want to be able to predict whether or not a customer will successfully complete an offer or not. Since the output is either a 1 or 0, this is a classification problem that we would like to solve.

While selecting the features needed for the model, I decided to change the gender column to numerical so that I could include it in the model. So the features I ended up using for the model were age, bogo, discount, duration, email, gender, income, informational, member year, member month, mobile, reward, social, time, and web. And of course, our predictor here is the offer_successful column. Once we have our features and predictor ready, we will split the data into training and testing.

Now for creating the model. Let’s start with creating the Logistic Regression model. We will create two of these. One model will run on the data that is only standardized, the second will run on the data that is standardized then normalized. The results of these two will be compared.

Let’s tune both the Logistic Regression and the Random Forest models, and then compare the two.

Hyperparameter tuning

For tuning the Logistic Regression model, we added values of C as our tuning parameter. Here, C is the inverse of regularization strength. So, the smaller values of C indicate a stronger regularization. (Note) Regularization is the technique that discourages learning a more complex model in order to avoid overfitting.

To tune the Random Forest Classifier, we add a range of values for our estimators and a range of how many features we desire to be tested out. In this case, I used a range from 6 to 14 features (our max amount of features).
